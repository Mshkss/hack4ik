# Проект системы MVP: цифровой штурман аэролодки

## Цель MVP

Собрать рабочий прототип, который строит маршрут для аэролодки Raptor 650 не как обычную линию на карте, а как расчетное решение с учетом поверхности, топлива, риска, загрузки и режима поддува.

MVP должен показывать экспертам:

- карта и данные реально влияют на маршрут;
- разные режимы дают разные решения;
- параметры Raptor 650 участвуют в расчете;
- маршрут объясним: видно, почему он выбран;
- решение может работать офлайн на локальных данных.

## Границы MVP

### Входит

- выбор стартовой и конечной точки;
- демонстрационная карта с типами поверхности;
- граф маршрутов поверх карты;
- расчет маршрута через A*;
- режимы: короткий, быстрый, экономичный, безопасный;
- переключатель базовой конфигурации / поддува;
- расчет длины, времени, расхода, остатка топлива и риска;
- предупреждения о мели, неизвестных зонах и нехватке топлива;
- карточка объяснения маршрута;
- экспорт или хранение результата в GeoJSON/JSON.

### Не входит

- реальное управление лодкой;
- автопилот;
- юридически точная морская навигация;
- подключение настоящей NMEA 2000 сети;
- точная физическая модель аэродинамики и глиссирования;
- гарантированная безопасность маршрута без проверки оператором.

## Основной пользовательский сценарий

1. Пользователь открывает карту.
2. Выбирает старт и финиш.
3. Указывает параметры: топливо, загрузка, поддув, режим маршрута.
4. Система строит несколько вариантов или выбранный режим.
5. Пользователь видит маршрут, карточку показателей и объяснение.
6. При изменении параметров маршрут пересчитывается.

## Архитектура MVP

```text
Data layer
  -> demo_area.geojson
  -> route_graph.geojson
  -> hazards.geojson
  -> surface_costs.json
  -> boat_profile_raptor_650.json
  -> test_scenarios.json

Domain layer
  -> BoatProfile
  -> BoatState
  -> SurfaceModel
  -> RouteGraph
  -> RouteCostModel

Planning layer
  -> AStarPlanner
  -> RouteEvaluator
  -> AlternativeRoutes

Explanation layer
  -> route summary
  -> segment reasons
  -> warnings
  -> data quality notes

UI layer
  -> map
  -> route controls
  -> route card
  -> comparison table
```

## Модули

| Модуль | Назначение | Вход | Выход |
| --- | --- | --- | --- |
| `MapDataLoader` | загружает локальные GeoJSON/JSON | файлы из `data/seed` | нормализованные слои |
| `RouteGraphBuilder` | готовит граф для A* | `route_graph.geojson` | узлы и ребра |
| `BoatProfileService` | хранит параметры Raptor 650 | `boat_profile_raptor_650.json` | профиль лодки |
| `BoatStateService` | хранит состояние поездки | топливо, загрузка, режим | `BoatState` |
| `SurfaceCostModel` | дает коэффициенты по поверхностям | `surface_costs.json` | стоимость участка |
| `RouteCostModel` | считает время, топливо, риск | ребро + состояние лодки | вес ребра |
| `AStarPlanner` | ищет маршрут | граф + start + finish + cost model | путь |
| `RouteExplainer` | объясняет выбор | путь + метрики | карточка объяснения |
| `DataQualityReporter` | показывает доверие к данным | метаданные источников | предупреждения |

## Поток данных

```text
Local files
  -> validation
  -> domain models
  -> graph
  -> A* route calculation
  -> route metrics
  -> explanation
  -> UI
```

Критично: расчетное ядро не должно зависеть от интерфейса. UI может быть заменен, а данные позже можно брать из Signal K, GPSD, NMEA или реальной карты.

## Внутренние сущности

### BoatProfile

Профиль судна из постановки и допущений MVP:

- модель;
- двигатель;
- бак;
- базовый расход;
- грузоподъемность;
- вместимость;
- конфигурации;
- коэффициенты поддува;
- безопасный резерв топлива.

### BoatState

Состояние конкретного расчета:

- остаток топлива;
- загрузка;
- включен ли поддув;
- выбранный режим маршрута;
- текущая позиция, если есть;
- качество данных.

### SurfaceSegment

Участок карты или ребра:

- тип поверхности;
- длина;
- риск;
- глубина или класс глубины;
- требуется ли поддув;
- можно ли проходить без поддува;
- источник данных.

### RouteResult

Результат расчета:

- список сегментов;
- общая длина;
- время;
- расход;
- остаток топлива;
- риск;
- предупреждения;
- объяснение.

## Алгоритм MVP

Для первой версии используется A* по заранее подготовленному графу маршрутов.

Причины:

- быстро реализуется;
- работает офлайн;
- легко объясняется;
- хорошо подходит для демонстрационных данных;
- позволяет менять веса под разные режимы.

Функция стоимости:

```text
edge_cost =
  mode.time_weight * time_cost
  + mode.fuel_weight * fuel_cost
  + mode.risk_weight * risk_cost
  + surface_penalty
  + unknown_data_penalty
  + blower_required_penalty
```

Если участок недоступен без поддува, а поддув выключен, ребро исключается из графа или получает запретную стоимость.

## Режимы маршрута

| Режим | Что оптимизирует | Вес времени | Вес топлива | Вес риска |
| --- | --- | ---: | ---: | ---: |
| `shortest` | длину | 0.4 | 0.2 | 0.4 |
| `fastest` | время | 1.0 | 0.3 | 0.5 |
| `economy` | топливо | 0.4 | 1.0 | 0.6 |
| `safe` | безопасность | 0.3 | 0.5 | 1.0 |

Значения являются стартовыми MVP-коэффициентами. Их нужно калибровать после экспертной проверки.

## Интерфейсы на будущее

| Интерфейс | Роль после MVP | Что меняется |
| --- | --- | --- |
| Signal K | источник бортовых данных | `BoatState` наполняется из API |
| GPSD | текущая позиция | старт можно брать от GPS |
| NMEA 0183 | GPS/depth/AIS | добавляется адаптер входных сообщений |
| NMEA 2000/CANboat | датчики и двигатель | добавляется bridge телеметрии |
| GPX | обмен маршрутами | маршрут можно отдавать в OpenCPN |
| GeoPackage/MBTiles | офлайн-карты | заменяет часть GeoJSON-слоев |

## Минимальный технический результат

К концу MVP должны быть:

1. Локальные данные в `data/seed`.
2. Документированный формат данных.
3. Расчетный граф маршрутов.
4. Профиль Raptor 650.
5. Набор тестовых сценариев.
6. Реализованный расчет маршрута.
7. Визуализация маршрута и карточка объяснения.

## Ключевые риски

| Риск | Что делаем в MVP |
| --- | --- |
| Нет точных полевых данных | явно маркируем seed-данные как демонстрационные |
| Нет реальной NMEA сети | проектируем адаптеры, но используем JSON |
| Нет официальных глубин | используем слой риска и ручную разметку |
| Модель расхода приблизительная | показываем диапазон и коэффициенты |
| Пользователь может принять демо за навигационный продукт | добавляем предупреждение "не для реальной навигации" |
