# Данные для реализации MVP

## Принцип

Данные MVP делятся на три уровня:

1. **Обязательные seed-данные.** Нужны сразу, чтобы прототип работал без внешних сервисов.
2. **Открытые внешние данные.** Подключаются для расширения карты и проверки реальности.
3. **Полевые данные.** Собираются позже на реальной аэролодке или от экспертов.

Все данные должны хранить признак происхождения: `case`, `manual_seed`, `open_source`, `field_measurement`, `assumption`.

## Схема GeoJSON-слоев

### `demo_area.geojson`

Описывает демонстрационную область:

- `id`;
- `name`;
- `purpose`;
- `bbox`;
- `data_status`;
- `navigation_grade`.

### `route_graph.geojson`

Содержит точки и ребра графа.

Для узлов:

```json
{
  "kind": "node",
  "node_id": "start_base",
  "label": "База",
  "allowed_as_start": true,
  "allowed_as_finish": true
}
```

Для ребер:

```json
{
  "kind": "edge",
  "edge_id": "e01",
  "from": "start_base",
  "to": "main_channel",
  "surface": "deep_water",
  "distance_km": 4.2,
  "risk": 0.18,
  "requires_blower": false,
  "source": "manual_seed"
}
```

### `hazards.geojson`

Описывает рисковые зоны:

- `hazard_type`: `shallow`, `reed`, `mud`, `blocked`, `unknown`;
- `risk_delta`;
- `recommended_action`;
- `source`;
- `confidence`.

## Нужные внешние данные

| Категория | Источник | Приоритет | Что собираем |
| --- | --- | ---: | --- |
| Береговая линия и водные объекты | OpenStreetMap / Overpass | P0 | реки, озера, острова, берег |
| Водная инфраструктура | OpenStreetMap / OpenSeaMap | P1 | причалы, переправы, знаки, точки интереса |
| Форматы и лицензии | GeoJSON RFC, OSM copyright | P0 | правила хранения и атрибуции |
| Офлайн-контейнер | OGC GeoPackage | P2 | следующий формат после GeoJSON |
| Официальные ENC | IHO/NOAA как референс | P2 | структура навигационных слоев |
| Бортовые данные | Signal K/GPSD | P2 | будущий live-вход координат и телеметрии |

## Полевые данные после MVP

| Данные | Как собирать | Зачем |
| --- | --- | --- |
| Реальный расход по поверхностям | треки + заправки + режим | калибровка топлива |
| Реальная скорость | GPS-трек | ETA и глиссирование |
| Точки мелей | экспертная разметка/эхолот | риск |
| Проходимость камыша/грязи | отметки оператора | поддув и запреты |
| Потеря глиссирования | скорость + загрузка + заметки | штрафы модели |
| Сезонность | дата + состояние воды/льда | надежность карты |

## Требования к качеству данных

Каждая сущность должна иметь:

- `source`;
- `confidence`;
- `updated_at`;
- `navigation_grade`;
- `notes`.

Для seed-данных `navigation_grade` всегда `false`.

## Минимальный контроль перед использованием

- все GeoJSON-файлы являются `FeatureCollection`;
- все координаты в порядке `[lon, lat]`;
- все ребра графа имеют `from`, `to`, `surface`, `distance_km`;
- все `from`/`to` ссылаются на существующие узлы;
- все поверхности есть в `surface_costs.json`;
- все сценарии используют существующие узлы;
- профиль лодки содержит бак, расход, груз и конфигурации.

## Что означает "начат сбор данных"

В текущей итерации сбор данных начат так:

1. Создана структура `data`.
2. Создан стартовый seed-набор.
3. Зафиксированы источники открытых данных.
4. Подготовлена схема, в которую можно импортировать OSM/Overpass и будущие полевые данные.
5. Данные отделены от кода и могут быть подключены в MVP без переписывания архитектуры.
