# Техническое задание для ИИ-агента

## 1. Назначение документа

Этот документ является рабочей памяткой для дальнейшей разработки проекта "Кубок Енисея 2026". Его нужно использовать как техническое задание, план работ и общее архитектурное направление для ИИ-агента.

Документ опирается на:

- `docs/context.md` - постановка задачи и требования к цифровому штурману;
- `docs/критерии оценки.md` - критерии, по которым будет оцениваться MVP;
- `docs/синтетические_данные.md` - пример локальной модели данных;
- обсужденную архитектурную позицию команды: web UI нужен только для демонстрации, итоговое решение должно быть расчетным offline-first модулем для бортового контура.

Главная идея: реализовать не "сайт с маршрутом", а автономное расчетное ядро цифрового штурмана для аэролодки. Временный web-интерфейс обязателен для демонстрации MVP экспертам, но не должен содержать основную бизнес-логику.

## 2. Суть продукта

Продукт - цифровой штурман для аэролодки Raptor 650. Он помогает экипажу заранее выбрать маршрут, оценить время, расход топлива, риски, запас хода и вероятность потери глиссирования.

Система должна работать offline-first:

- карта хранится локально;
- разметка поверхности хранится локально;
- параметры лодки хранятся локально;
- коэффициенты расхода и риска хранятся локально;
- расчет маршрута выполняется локально;
- базовые сценарии не зависят от облака, интернета и внешних API.

Приложение не управляет лодкой напрямую. Оно является системой поддержки принятия решений.

## 3. Важное архитектурное решение

Web UI не является основным продуктовым слоем итоговой системы. Он нужен как обязательная демонстрационная оболочка MVP для проверки сценариев экспертами:

- загрузка карты;
- выбор лодки;
- выбор конфигурации Raptor 650;
- выбор загрузки и запаса топлива;
- построение маршрута;
- смена режима движения;
- расчет топлива, времени и рисков;
- предупреждения;
- итоговая карточка маршрута.

Итоговое решение должно быть построено вокруг расчетного ядра и локальной модели данных. UI должен быть заменяемым, но demo UI должен сохраняться в репозитории и оставаться доступным для проверки после хакатона.

Правильная формулировка для защиты:

> Основной продукт - автономный расчетный модуль цифрового штурмана. Web UI используется как обязательный демонстрационный слой MVP и может быть заменен на интерфейс бортового терминала или интеграцию с внешним навигационным устройством.

## 4. Целевая бортовая схема

Предполагаемый будущий контур:

```text
Simrad GO9 XSE
  |
NMEA2000 сеть
  |
NMEA2000 USB/CAN gateway
  |
Внешний Linux-модуль: Raspberry Pi / industrial SBC / mini PC
  |
Расчетный модуль цифрового штурмана
  |
Локальный UI / web demo / API / журналирование
```

Важно: в системе не предполагается сразу несколько вычислительных плат. Ниже перечислены не обязательные устройства, а классы железа и их роли.

Основной вычислительный модуль должен быть один. Для MVP и будущего борта это может быть Raspberry Pi, промышленный Linux SBC или mini PC. Raspberry Pi здесь используется как понятный пример Linux-компьютера, а не как жесткое требование.

Такой модуль выбран для основной логики, потому что цифровому штурману нужны:

- локальное файловое хранилище;
- запуск расчетного ядра;
- работа с JSON/GeoJSON/SQLite;
- чтение данных через USB/CAN gateway;
- Wi-Fi для демонстрационного интерфейса или локальной панели;
- логирование;
- автозапуск;
- возможность замены Raspberry Pi на промышленный SBC без смены архитектуры.

ESP32 или Arduino - это не альтернатива основному вычислителю для этой задачи. Это примеры микроконтроллеров, которые можно добавить только как опциональную периферию. Их допустимая роль:

- периферийный датчик;
- простой CAN-узел;
- Wi-Fi/CAN мост;
- сбор отдельных телеметрических параметров.

Они не должны хранить карту, строить маршруты и выполнять основную расчетную логику.

## 5. Рекомендуемый стек

Так как сложность реализации не является главным ограничением, лучший вариант для убедительной архитектуры:

```text
C++20 расчетное ядро
Python scripts для подготовки данных, генерации сценариев и тестов
  временный Web UI для обязательной демонстрации MVP
целевая платформа: Raspberry Pi / Linux SBC
будущая интеграция: NMEA2000 через USB/CAN gateway
```

Для MVP использовать практичный минимальный набор:

```text
C++20 core:
  nlohmann/json
  собственная реализация Dijkstra
  CMake

Python / desktop tools:
  QGIS для подготовки карты и ручной разметки
  Python-скрипты для валидации JSON/GeoJSON

Demo UI:
  Leaflet
  локальная PNG/MBTiles-подложка
  GeoJSON-слои поверх карты
```

Расширяемые библиотеки, которые можно добавить после базового MVP:

```text
GeographicLib      расчет расстояний по координатам
Boost.Geometry     работа с полигонами и пересечениями
Boost.Graph        графовые алгоритмы, если своего Dijkstra станет мало
spdlog             журналирование
Catch2 / GoogleTest тесты C++ ядра
GDAL / rasterio    подготовка растровых карт
geopandas / shapely подготовка и проверка GeoJSON
MapLibre GL JS     более продвинутая карта вместо Leaflet
```

### 5.1. Почему C++20 для ядра

C++20 хорошо подходит для расчетного ядра, потому что:

- переносим на x86 и ARM Linux;
- выглядит убедительно для будущего бортового контура;
- позволяет собрать автономный бинарник;
- не требует тяжелого runtime в финальной версии;
- подходит для маршрутизации, расчетов, оценки рисков и работы с локальными данными;
- лучше отделяет инженерную часть от временного web UI.

### 5.2. Где использовать Python

Python использовать как вспомогательный слой:

- подготовка карты;
- генерация синтетических сценариев;
- валидация JSON/GeoJSON;
- тестовые прогоны;
- построение отчетов по маршрутам;
- при необходимости - временный wrapper вокруг C++ CLI для demo API.

Python не должен быть обязательной частью финального бортового расчетного ядра, если ядро реализовано на C++.

### 5.3. Где использовать C

Чистый C не нужен как backend. Возможная роль C в будущем:

- низкоуровневые адаптеры;
- работа с CAN;
- интеграция с существующими C-библиотеками для NMEA2000.

Основной backend или расчетное ядро на чистом C делать нецелесообразно.

### 5.4. Web UI

Web UI делать как съемный, но обязательный для MVP демонстрационный слой:

- можно использовать Vite + React + TypeScript;
- можно использовать Leaflet или MapLibre для карты;
- UI должен вызывать расчетное ядро через локальный API или CLI wrapper;
- бизнес-логика не должна жить во frontend;
- после демонстрации UI можно заменить без потери основного продукта, но в рамках оценки он должен оставаться доступным.

## 7. Расчетное ядро

Расчетное ядро должно быть независимым от интерфейса. Оно получает локальные входные данные и возвращает структурированный результат.

Минимальный вход:

- карта;
- стартовая точка;
- конечная точка;
- модель лодки;
- конфигурация лодки: без поддува или с поддувом;
- загрузка;
- запас топлива;
- режим маршрута;
- локальные коэффициенты поверхностей.

Минимальный выход:

- найденный маршрут;
- список сегментов маршрута;
- длина маршрута;
- расчетное время;
- расчетный расход топлива;
- остаток топлива;
- запас хода;
- средний риск;
- максимальный риск;
- участки с потерей глиссирования;
- предупреждения;
- краткое объяснение выбора маршрута.

## 8. Модель данных

За основу брать `docs/синтетические_данные.md`.

Текущий пример уже содержит важные сущности:

- `scenario` - описание демонстрационного сценария;
- `boat` - параметры Raptor 650;
- `configs` - варианты без поддува и с поддувом;
- `surfaces` - типы поверхностей;
- `modes` - режимы маршрута;
- `formulas` - описание расчетов;
- `map.edges` - граф маршрутов.

Перед реализацией данные желательно нормализовать.

Для машинной обработки использовать стабильные ID на английском:

```json
    "economy": {

      "objective": "fuel",

    },
    "short": {
      "label": "кратчайший",
      "objective": "length",
```

### 8.2. Единицы измерения

Поля должны явно содержать единицы:

- `speed_kmh` вместо `spd`;
- `length_km` вместо `km`;
- `tank_liters` вместо `tank_l`;
- `base_l_per_km` оставить как есть;
- `reserve_fraction` или `reserve_frac_tank`.

### 8.3. Узлы карты

Для web demo и будущей визуализации нужны координаты узлов:

```json
{
  "nodes": {
    "divnogorsk": {
      "label": "Дивногорск",
      "lat": 55.95,
      "lon": 92.37
    },
    "biryusa": {
      "label": "Бирюса",
      "lat": 55.91,
      "lon": 92.65
    }
  }
}
```

Ребра должны ссылаться на ID, а не на отображаемые названия:

```json
{
  "from": "divnogorsk",
  "to": "polynya",
  "length_km": 8,
  "surface": "water"
}
```

### 8.4. Карта для MVP

Для MVP выбран подход с одним подготовленным регионом, а не попытка покрыть весь мир.

MVP-регион:

```text
Красноярский край / участок Енисея / выбранная акватория для демонстрации
```

Причина: критерии требуют показать работающий прототип, реальную карту местности, разметку поверхностей и маршруты, которые реально меняются от условий. Это надежнее сделать на одном качественно подготовленном участке. Синтетическими могут быть расчетные слои, коэффициенты, зоны риска и демонстрационные сценарии, но визуальная карта должна быть привязана к реальной местности.

Карта должна состоять из двух разных слоев:

```text
1. Визуальная подложка
   локальная PNG-карта или MBTiles-тайлы.
   Нужна пользователю для ориентации.

2. Расчетный слой
   routing_graph.geojson
   hazard_zones.geojson
   surfaces.json
   Нужен алгоритму для построения маршрута и расчетов.
```

Маршрут не должен считаться по картинке. Картинка только отображает местность. Расчет выполняется по графу, зонам и коэффициентам.

Связь `hazard_zones.geojson` с графом должна быть явной. При загрузке карты система должна применить зоны риска к ребрам графа:

```text
если ребро целиком лежит в зоне -> ребро получает surface/risk/penalty этой зоны
если ребро пересекает зону -> ребро делится на сегменты или получает долевой штраф
если ручная разметка уже указала surface на ребре -> зона может усилить risk или добавить warning
```

Для MVP допустим простой вариант: каждое ребро в `routing_graph.geojson` уже имеет поле `surface`, а `hazard_zones.geojson` используется для визуализации и дополнительных предупреждений. Но в документации нужно явно показать, что эти зоны соответствуют ребрам и влияют на расчет, а не являются декоративной заливкой на карте.

Пользователь в demo UI выбирает точки A и B кликом по карте. Система должна:

```text
клик A -> найти ближайший допустимый узел или ребро графа
клик B -> найти ближайший допустимый узел или ребро графа
запустить Dijkstra по выбранному режиму
вернуть маршрут и эксплуатационные показатели
показать маршрут и предупреждения на карте
```

Для подготовки MVP-карты использовать QGIS:

```text
1. Взять подложку нужного участка.
2. Геопривязать ее, если это PNG/скрин.
3. Нарисовать граф проходов по воде/льду/мелководью.
4. Нарисовать зоны риска: болото, камни, шуга, мелководье, сложные участки.
5. Экспортировать routing_graph.geojson и hazard_zones.geojson.
6. Проверить, что каждый участок связан с surface из surfaces.json.
```

OpenStreetMap можно использовать как источник геоданных или подложки, но не нужно встраивать сайт `openstreetmap.org` как iframe. Если нужен вид обычной карты без лишних баннеров и чужих элементов интерфейса, правильный путь для MVP:

```text
локальная PNG-подложка из QGIS
или локальные MBTiles
или собственный стиль тайлов
```

Если используются данные OpenStreetMap, нужно оставить нейтральную атрибуцию вида:

```text
© OpenStreetMap contributors
```

Флаг, баннеры и элементы чужого сайта не являются обязательной частью карты, если не использовать их сайт или конкретный внешний tile-provider как готовый виджет.

### 8.5. Масштабирование после MVP

Алгоритм должен быть написан так, чтобы не зависеть от Красноярского края. Но данные для всего мира не нужно делать в MVP.

Правильная масштабируемая модель:

```text
data/regions/
  krasnoyarsk_mvp/
    base_map.mbtiles
    routing_graph.geojson
    hazard_zones.geojson
    surfaces.json
    scenario.json

  baikal_future/
    base_map.mbtiles
    routing_graph.geojson
    hazard_zones.geojson
    surfaces.json
    scenario.json

  any_region_future/
    ...
```

То есть ядро работает с любым региональным пакетом, если в нем есть:

- визуальная карта;
- граф маршрутизации;
- зоны риска;
- справочник поверхностей;
- сценарий или входные параметры.

Формулировка для защиты:

> MVP работает на одном подготовленном участке Енисея. Алгоритм не привязан к региону: для новой акватории подключается новый локальный пакет данных с картой, графом, зонами риска и коэффициентами поверхностей.

## 9. Маршрутизация

Карта на первом этапе может быть графом:

- вершина - точка маршрута;
- ребро - участок между точками;
- у ребра есть длина, тип поверхности и расчетные параметры через `surfaces`.

Алгоритм:

- Dijkstra для режима без эвристики;
- A* можно добавить, если у узлов есть координаты;
- для MVP Dijkstra достаточно.

Каждый режим должен менять вес ребра. Переключатели режимов не должны быть декоративными.

Примеры целевых функций:

```text
fast:
  weight = time_hours + risk_penalty + hard_penalty

economy:
  weight = fuel_liters + risk_penalty + hard_penalty

short:
  weight = length_km + impassable_penalty

safe:
  weight = length_km + high_risk_penalty + no_planing_penalty + hard_penalty

planing:
  weight = time_hours + no_planing_penalty + fuel_liters * small_factor
```

Если для выбранной конфигурации лодки участок `hard = true`, а `allow_hard = false`, участок должен считаться непроходимым или получать запретительно высокий штраф. Лучше для MVP считать его непроходимым и показывать предупреждение.

## 10. Расчеты

Расчеты должны быть простыми, объяснимыми и связанными с данными.

Базовые формулы:

```text
time_segment_hours = length_km / speed_kmh

fuel_segment_liters =
  length_km
  * base_l_per_km
  * surface.k_surf
  * config.k_load
  * mode.k_mode
  * payload_factor

fuel_total_liters = sum(fuel_segment_liters)

time_total_hours = sum(time_segment_hours)

fuel_remaining_liters = fuel_start_liters - fuel_total_liters

average_l_per_km = fuel_total_liters / total_length_km

range_remaining_km = fuel_remaining_liters / average_l_per_km
```

Фактор загрузки должен быть определен явно. Для MVP использовать простую линейную модель:

```text
payload_factor = 1.0 + 0.20 * clamp(payload_kg / 1500, 0, 1)
```

Где `1500 кг` - верхняя оценка грузоподъемности Raptor 650 из постановки. Такая модель означает, что максимальная загрузка увеличивает расход примерно на 20%. Коэффициент можно изменить в данных, но он должен быть виден в документации и тестовых сценариях.

Резерв топлива:

```text
reserve_liters = boat.tank_liters * reserve_frac_tank
```

Если после маршрута остаток меньше резерва, система должна выдать предупреждение. Маршрут не должен считаться безопасным, если топливо заканчивается ровно в конечной точке.

## 11. Предупреждения и объяснимость

Предупреждения являются важной частью оценки. Система должна не только строить маршрут, но и объяснять результат.

Минимальные предупреждения:

- недостаточный запас топлива;
- остаток топлива ниже эксплуатационного резерва;
- участок с высоким риском;
- участок с повышенным расходом;
- возможная потеря глиссирования;
- маршрут проходит через сложную поверхность;
- участок недоступен для лодки без поддува;
- выбран более длинный маршрут из-за риска или расхода.

Пример объяснения:

```text
Безопасный режим выбрал более длинный путь через лед, потому что короткий путь проходит через камни и болото. Для конфигурации без поддува hard-участки запрещены, а риск и расход на них выше.
```

## 12. Демонстрационные сценарии

Сценарии должны напрямую соответствовать критериям оценки.

### Сценарий 1. Подготовка карты и автономная работа

Показать, что:

- карта загружена локально;
- разметка поверхностей доступна;
- параметры Raptor 650 доступны;
- маршруты считаются без интернета;
- данные лежат в локальных файлах.

### Сценарий 2. Построение маршрута из точки А в точку Б

Пользователь выбирает:

- старт;
- финиш;
- вариант Raptor 650;
- загрузку;
- запас топлива;
- режим движения.

Система показывает:

- маршрут;
- длину;
- время;
- расход;
- остаток топлива;
- запас хода;
- риск.

### Сценарий 3. Сравнение режимов

Для одной пары точек построить:

- быстрый;
- экономичный;
- кратчайший;
- безопасный;
- с поддержанием глиссирования.

Нужно добиться, чтобы хотя бы часть маршрутов отличалась. Если маршруты совпадают, система должна объяснить почему.

### Сценарий 4. Сложный участок и предупреждения

Показать маршрут через:

- мелководье;
- шугу;
- траву;
- камни;
- болото;
- смешанную зону, если она есть в данных.

Система должна показать влияние на:

- скорость;
- расход;
- риск;
- глиссирование;
- выбор маршрута.

### Сценарий 5. Конфигурация лодки и загрузка

Сравнить:

- Raptor 650 без поддува;
- Raptor 650 с поддувом;
- разную загрузку.

Хороший результат: участок, невозможный или нежелательный без поддува, становится допустимым с поддувом, но меняет расход или риск.

### Сценарий 6. Карточка маршрута

Итоговая карточка должна содержать:

- карту или схему прохождения;
- режим;
- длину;
- время;
- расход;
- остаток топлива;
- запас хода;
- рисковые участки;
- предупреждения;
- краткое объяснение выбора маршрута.

## 13. Приоритеты разработки по критериям оценки

Максимальный эффект на оценку дают:

1. Реальная или правдоподобная карта с разметкой, которая реально используется в расчете.
2. Режимы маршрута, которые действительно меняют вес ребер и результат.
3. Эксплуатационные расчеты, связанные с поверхностью, загрузкой, поддувом, топливом и резервом.
4. Предупреждения и объяснимость.
5. Offline-first: локальные данные и запуск без облака.
6. Модульная архитектура с отделением UI от расчетного ядра.
7. Документация и воспроизводимый запуск.
8. Отсутствие зависимости от платного или проприетарного ПО, без которого нельзя проверить прототип.

## 14. Что не делать

Не делать:

- не строить продукт вокруг web UI;
- не хранить бизнес-логику во frontend;
- не делать режимы декоративными переключателями;
- не использовать внешние облачные API как обязательную часть расчета;
- не делать ESP32 или Arduino главным вычислителем;
- не делать backend на чистом C без необходимости;
- не смешивать подготовку данных, расчет маршрута и визуализацию в одном файле;
- не завязывать решение на платное или проприетарное ПО;
- не считать маршрут безопасным, если топливо заканчивается впритык.

## 15. Минимальный MVP

Минимальный MVP должен уметь:

- работать на одном подготовленном регионе `krasnoyarsk_mvp`;
- показывать локальную карту через PNG-подложку или MBTiles;
- отображать расчетные GeoJSON-слои: граф маршрутов и зоны риска;
- принимать выбор точек A и B на карте и привязывать их к ближайшим допустимым точкам графа;
- читать локальный JSON с картой, поверхностями, лодкой и режимами;
- строить граф;
- считать маршрут Dijkstra по выбранному режиму;
- строить и сравнивать маршруты для всех режимов: быстрый, экономичный, кратчайший, безопасный, с поддержанием глиссирования;
- учитывать hard-участки и конфигурацию лодки;
- считать длину, время, расход, остаток топлива, запас хода;
- фиксировать потерю глиссирования;
- формировать предупреждения;
- возвращать результат в JSON;
- сохранять рассчитанный маршрут или демо-сценарий локально;
- запускаться локально по инструкции;
- показывать результат во временном demo UI;
- иметь CLI как дополнительный способ проверки расчетного ядра без UI.

## 16. Расширенный MVP

Если базовый MVP готов, добавить:

- более точные координаты узлов и геометрию ребер;
- отдельные стили отображения для поверхностей и рисков;
- несколько дополнительных демонстрационных сценариев;
- пакетный расчет нескольких сценариев одним запуском;
- журналирование расчетов;
- симулятор NMEA2000-данных;
- интерфейс adapter layer для будущих GPS, топлива, двигателя, эхолота и лидара.

## 17. Рекомендуемый CLI-контракт

Расчетное ядро должно уметь запускаться без UI:

```text
navigator_cli calculate --input data/scenarios/demo.json --output out/route_result.json
```

Пример результата:

```json
{

    "mode": "safe",
    "nodes": ["divnogorsk", "ice_1", "junction_m", "clean", "biryusa"],
    "total_length_km": 46.0,
    "total_time_hours": 0.73,
    "fuel_used_liters": 34.2,
    "fuel_remaining_liters": 265.8,
```

CLI важен, потому что он доказывает независимость расчетного ядра от web UI.

## 18. Интеграция с NMEA2000

Для MVP не требуется реальная работа с Simrad GO9 XSE и NMEA2000. Нужно подготовить архитектуру:

```text
adapters/
  nmea2000_simulator
  nmea2000_gateway
```

На первом этапе интеграция должна быть read-only:

- GPS/ГНСС;
- скорость;
- курс;
- топливо;
- двигатель;
- другие доступные параметры.

Запись в NMEA2000-сеть не делать в MVP. Это отдельная задача из-за безопасности, совместимости PGN и поведения устройств в сети.

## 20. Итоговая стратегия

Основная стратегия:

```text
Сначала сделать сильное расчетное ядро и локальные данные.
Потом сделать временную демонстрацию.
После этого оформить архитектуру и сценарии так, чтобы они напрямую закрывали критерии оценки.
```

Лучшее исполнение для проекта:

```text
C++20 core
CLI с JSON input/output
локальные JSON/GeoJSON данные
Python tools для подготовки и проверки
временный web UI для демонстрации
целевая платформа Raspberry Pi / Linux SBC
будущая интеграция через NMEA2000 USB/CAN gateway
```

Ключевой тезис:

> В проекте оценивается не сайт, а способность системы автономно построить и объяснить маршрут аэролодки с учетом карты, поверхности, топлива, риска, режима движения и конфигурации Raptor 650.
