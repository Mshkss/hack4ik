# Сверка с ТЗ и план доработки

Дата сверки: 2026-06-27.

Источник требований: `Postanovka.pdf`, кейс "Цифровой штурман для аэролодки". PDF является сканом, требования сняты визуально со страниц 2-18.

## Суть кейса

Нужно сделать offline-first прототип цифрового штурмана для аэролодки Raptor 650. Пользователь выбирает старт, финиш, конфигурацию лодки, загрузку, запас топлива и приоритет маршрута. Система строит маршрут по реальной или заранее подготовленной карте, учитывает типы поверхностей, время, топливо, риски, возможность глиссирования и объясняет, почему выбран именно такой путь.

Решение не должно управлять лодкой. Это система поддержки решения экипажа: карта, расчет, предупреждения, сравнение вариантов и понятная карточка маршрута.

## Фичи, которые используем

### Реализованные

- `A*`-маршрутизация по локальному графу с эвристикой расстояния.
- C++ route engine как отдельный расчетный модуль.
- Node.js API-обертка для `/api/scenario` и `/api/route`.
- Автоматическая генерация графа из surface zones, OSM waterway lines и локальных seed-точек.
- Плотная водная сеть: узлы waterway через `0.08 км`, navmesh через `0.12 км`, межлинейные waterway-связки до `0.18 км`.
- Нейтральная карта без подписей и политических атрибутов.
- Локальные GeoJSON/route artifacts: `scenario.route`, `scenario_meta.json`, `route_graph_georef.geojson`, `surface_zones.geojson`.
- Выбор старта/финиша из списка, по координатам или кликом.
- Привязка пользовательской точки к ближайшему узлу графа.
- Пеший добор от/до воды, если точка стоит не на водном графе.
- Красный пунктир для направления, где водный проезд не найден.
- Режимы маршрута: быстрый, экономичный, кратчайший, безопасный.
- Конфигурации лодки: без поддува, с поддувом.
- Настройка параметров Raptor 650 и физической модели.
- Расчет скорости, времени, топлива, мощности, сопротивления, риска, резерва.
- Состояния движения: водоизмещающий режим, переходный режим, глиссирование.
- Рекомендации по скорости на участке: медленно, умеренно, можно быстрее.
- Кавитационный/срывной риск и предупреждения.
- Таблица поверхностей и статистика графа.
- Сравнение режимов в интерфейсе.
- Docker-контейнер для локального запуска.
- PDF с формулами.

### Фичи, которые стоит добавить

- `Offline Map Pack`: локальный пакет тайлов/векторной подложки для полной автономности.
- `Route Explain`: короткое объяснение, почему маршрут выбран, почему отвергнуты альтернативы и какие факторы дали самый большой вклад.
- `Scenario Launcher`: кнопки запуска сценариев из ТЗ без ручной настройки.
- `Data Freshness Badge`: статус данных: когда обновлены OSM, зоны, погода/лед/уровень воды, что доступно офлайн.
- `Water Reachability Inspector`: подсветка достижимой воды для выбранной конфигурации без построения маршрута.
- `Risk Heatmap`: тепловой слой риска по поверхностям и узким участкам.
- `Fuel Reserve Guard`: отдельная панель "доедем / не доедем" с 15/20/25% резервом.
- `Live Boat Profile`: сохранение нескольких профилей лодки и загрузки.
- `Telemetry Mock`: имитация GPS, топлива, оборотов, температуры двигателя, ветра и уровня воды.
- `Dynamic Obstacle Mode`: D* Lite для пересчета при закрытии участка, изменении льда или появлении запрета.
- `Hybrid Motion Mode`: Hybrid A* для учета радиуса поворота, ширины протоки и инерционности аэролодки.
- `Evidence Panel`: список источников данных, из которых получены вода, препятствия и поверхности.
- `Export Mission Pack`: экспорт маршрута, предупреждений и параметров в один JSON/PDF для экипажа.

## План доработки

### Приоритет 1: довести демо до уверенного показа

1. Добавить панель `Демо-сценарии` с 6 сценариями из ТЗ.
2. Сделать краткое объяснение выбора маршрута: топ-3 фактора стоимости и рисков.
3. Добавить локальную векторную подложку или offline fallback, чтобы карта не зависела от интернет-тайлов.
4. Сделать smoke-тесты: импорт, сборка C++, `/api/scenario`, `/api/route`, маршруты по новым точкам.
5. Подготовить скриншоты ключевых экранов и 7-минутный сценарий презентации.

### Приоритет 2: повысить достоверность карты

1. Расширить каталог мест из OSM `place=*` и ручного JSON.
2. Добавить отдельный слой водных глубин/мелей, если удастся найти открытый источник.
3. Добавить ручной редактор/импорт зон риска: лед, шуга, камни, болото, запрет.
4. Показать качество данных: `demo`, `estimated`, `verified`, `navigation-grade=false`.
5. Сделать инструмент проверки связности графа и узких рек.

### Приоритет 3: подготовка к бортовому контуру

1. Ввести интерфейс `BoatTelemetry`: GNSS, курс, скорость, топливо, обороты, температура, аварии.
2. Добавить mock-адаптеры NMEA 0183/2000, CAN, CSV replay.
3. Вести журнал расчетов и ошибок.
4. Описать формат mission pack для передачи маршрута на бортовой терминал.
5. Разделить C++ движок на библиотечный слой и CLI-обертку.

### Приоритет 4: алгоритмическое развитие

1. Оставить A* как production-базу MVP.
2. Добавить D* Lite для динамических изменений графа.
3. Добавить Hybrid A* для ограничений маневра в узких протоках.
4. Проверить RRT* для непрерывных зон без готового графа.
5. Добавить multi-objective сравнение маршрутов как объяснение, но не усложнять основной UX.
