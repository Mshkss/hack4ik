# 10. Критерии качества решения

## Зачем этот блок

Критерии нужны, чтобы команда могла оценивать не только внешний вид прототипа, но и соответствие кейсу. Хороший результат должен быть понятен экспертам, запускаться локально и демонстрировать расчётную логику.

## Критерии MVP

| Критерий | Как проверить |
| --- | --- |
| Offline-first | приложение строит маршрут без сети |
| Карта влияет на расчёты | изменение поверхности меняет маршрут/расход/риск |
| Режимы отличаются | быстрый, экономичный, кратчайший и безопасный дают разные показатели |
| Параметры Raptor 650 используются | изменение топлива, загрузки или поддува меняет результат |
| Есть топливный резерв | система предупреждает при нехватке резерва |
| Есть риск-модель | риск считается и показывается по участкам |
| Есть глиссирование | маршрут учитывает сохранение/потерю режима |
| Есть объяснимость | пользователь видит, почему выбран маршрут |
| Есть карточка маршрута | итоги собраны в понятном виде |
| Есть архитектурная схема | видно, как перенести логику на бортовой терминал |

## Критерии документации

В README или отдельной документации должны быть:

- инструкция по запуску;
- описание используемых данных;
- формат карты;
- параметры Raptor 650;
- таблица коэффициентов;
- описание расчёта времени, топлива, риска и глиссирования;
- описание режимов маршрута;
- скриншоты или описание ключевых экранов;
- схема архитектуры;
- ограничения модели.

## Критерии демонстрационных данных

Данные должны включать:

- минимум одну карту;
- минимум 5 типов поверхности;
- минимум одну непроходимую или условно проходимую зону;
- минимум одну зону высокого риска;
- минимум один маршрут, где безопасный путь длиннее быстрого;
- минимум один маршрут, где поддув меняет допустимость;
- минимум один сценарий с недостаточным топливным резервом.

## Ошибки, которых стоит избежать

- Карта есть, но расчёты от неё не зависят.
- Режимы маршрута меняют только подписи или цвет.
- Топливо считается только как `длина * расход`, без поверхности и загрузки.
- Нет объяснения, почему выбран маршрут.
- Нет предупреждений.
- Приложение требует интернет для базового сценария.
- Нельзя понять, какие коэффициенты использованы.
- UI перегружен таблицами и мелкими числами.

## Итоговая формула готовности

Прототип можно считать сильным, если эксперт за 2-3 минуты видит:

1. карта загружена локально;
2. маршрут построен автоматически;
3. режимы дают разные решения;
4. топливо, риск и глиссирование считаются;
5. система объясняет выбор;
6. архитектура готова к переносу на бортовой терминал.
