# 03. Карты и данные местности

## Зачем этот блок

Карта в кейсе должна быть источником расчётов, а не декоративной подложкой. Система должна понимать, какие участки доступны, какие поверхности там находятся, где повышенный риск и какие коэффициенты применить к скорости, расходу и глиссированию.

## Какие данные нужны

Минимальная модель карты:

| Слой | Что хранит | Зачем нужен |
| --- | --- | --- |
| `basemap` | визуальная карта/схема | ориентация пользователя |
| `surface_zones` | полигоны типов поверхности | коэффициенты скорости, расхода и риска |
| `blocked_zones` | непроходимые зоны | запрет маршрута |
| `risk_zones` | зоны повышенного риска | предупреждения и безопасный режим |
| `route_graph` | узлы и рёбра маршрутизации | построение пути |
| `test_routes` | сохранённые сценарии | демонстрация и проверка |

## Форматы данных

Для хакатонного MVP лучше использовать простые форматы:

| Формат | Когда использовать |
| --- | --- |
| GeoJSON | быстро описать полигоны, линии, точки и свойства |
| JSON | параметры лодки, коэффициенты, сценарии |
| SQLite/GeoPackage | если нужен более реалистичный offline-first контейнер |
| MBTiles/тайлы | если нужна локальная подложка карты |

GeoJSON хорош для старта: он основан на JSON и поддерживает `Point`, `LineString`, `Polygon`, `MultiPolygon`, `Feature` и `FeatureCollection`. Для прототипа это самый понятный способ хранить разметку поверхностей и маршрутов.

GeoPackage полезен как следующий шаг: это SQLite-контейнер для геоданных, рассчитанный на мобильные и offline-сценарии с ограниченной связью.

## Предлагаемая структура GeoJSON

```json
{
  "type": "FeatureCollection",
  "features": [
    {
      "type": "Feature",
      "geometry": {
        "type": "Polygon",
        "coordinates": []
      },
      "properties": {
        "surface": "shallow_water",
        "passable": true,
        "risk_level": 3,
        "speed_kmh": 22,
        "fuel_factor": 1.4,
        "planing_allowed": "conditional",
        "requires_blower": false
      }
    }
  ]
}
```

## Как связать карту с маршрутизацией

Есть два практичных подхода:

1. Граф маршрутов: команда вручную задаёт возможные рёбра между точками. Каждое ребро получает длину, поверхность, риск и коэффициенты. Это проще для MVP и хорошо объясняется экспертам.
2. Сетка: карта делится на клетки, а маршрут строится по соседним клеткам. Это гибче, но сложнее объяснить и отладить.

Для соревнования лучше начать с графа маршрутов. Он позволит явно показать, почему система выбрала один путь вместо другого.

## Демо-участок

Подойдёт участок реки/озера/смешанной территории, где есть:

- вода;
- мелководье;
- сложная поверхность;
- зона риска;
- альтернативный более длинный, но безопасный путь;
- участок, который допустим только с поддувом.
