# Готовые интерфейсы и функциональность для навигации по воде

## Что ищем

В кейсе цифрового штурмана важно понять, что уже можно взять готовым, а что придется делать самим. Готовые интерфейсы могут закрыть:

- отображение морских карт;
- импорт/экспорт маршрутов, waypoints и tracks;
- подключение GPS, AIS, глубины, курса и других датчиков;
- хранение карт офлайн;
- работу с морскими форматами;
- базовую навигационную UX-логику.

Но почти ни один готовый продукт не решит "из коробки" именно задачу аэролодки: маршрут с учетом нагнетателя, проходимости по мелководью/камышу/грязи, расхода топлива Raptor 650 и объяснимой оценки риска. Это нужно делать как собственное расчетное ядро.

## Карты и навигационные данные

### ENC, S-57, S-101, S-100

Электронные навигационные карты, ENC, являются базовым источником официальных морских данных: глубины, фарватеры, навигационные знаки, береговая линия, опасности. Исторически распространен стандарт S-57, а новое поколение гидрографических данных развивается вокруг S-100 и S-101.

Для российского/локального кейса доступность официальных данных может быть ограничением, но как архитектурный ориентир ENC важны. Они дают более правильную картографическую основу, чем просто спутниковая подложка.

Что можно взять:

- глубины и изобаты, если есть данные;
- береговую линию;
- навигационные опасности;
- фарватеры и зоны ограничений;
- структуру слоев для офлайн-карты.

Что не решают:

- проходимость аэролодки по камышу, грязи или суше;
- расход топлива;
- режим нагнетателя;
- локальные сезонные изменения.

### NOAA ENC

NOAA предоставляет открытые ENC для США. Для нашего проекта это полезный пример того, как официальные ENC могут поставляться и использоваться. Даже если демо не про США, NOAA ENC можно использовать как референс на тип данных и подход к навигационным слоям.

### OpenStreetMap и OpenSeaMap

OpenStreetMap и OpenSeaMap могут быть полезны для открытых геоданных: береговая линия, водные объекты, причалы, некоторые морские знаки и пользовательские слои. Для офлайн-прототипа эти данные можно предварительно выгрузить и преобразовать в GeoJSON/MBTiles/GeoPackage.

Ограничение: качество и полнота зависят от региона. Для безопасности нельзя слепо считать их официальной навигационной картой.

### GeoJSON, GeoPackage, MBTiles

Эти форматы удобны для MVP:

| Формат | Для чего использовать | Комментарий |
| --- | --- | --- |
| GeoJSON | зоны, ребра графа, точки, пользовательская разметка | прост и удобен для web |
| GeoPackage | офлайн-хранилище геоданных в SQLite | хорошо для мобильного/desktop offline |
| MBTiles | тайлы карты офлайн | удобно для быстрой отрисовки |
| GPX | обмен маршрутами и точками | поддерживается многими навигационными приложениями |

Для хакатона лучше начать с GeoJSON и GPX, а GeoPackage/MBTiles указать как следующий шаг для офлайн-продукта.

## Готовые приложения и платформы

### OpenCPN

OpenCPN - открытый картплоттер и навигационная платформа. Он поддерживает карты, маршруты, waypoints, tracks, работу с GPS/AIS/NMEA и расширение через плагины.

Что можно использовать:

- как референс UX для морской навигации;
- как внешний просмотрщик маршрута;
- импорт/экспорт GPX;
- подключение NMEA/Signal K данных;
- плагины и расширения, если проект пойдет в сторону интеграции.

Что OpenCPN не дает сразу:

- расчет маршрута под аэролодку с учетом нагнетателя;
- модель расхода Raptor 650;
- объяснение "почему выбран этот путь";
- сценарии "по воде/по мелководью/по камышу" в нужной нам логике.

Практический вывод: OpenCPN стоит рассматривать как готовый навигационный контур вокруг нашего расчетного ядра. Наше приложение может экспортировать GPX-маршрут, а OpenCPN использоваться для проверки и демонстрации в привычной морской среде.

### Signal K

Signal K - один из самых полезных готовых интерфейсов для проекта. Это не картплоттер, а открытый слой данных, который связывает морские датчики, приложения и web-интерфейсы.

Что можно использовать:

- прием данных с GPS/NMEA/NMEA 2000 через сервер;
- WebSocket/HTTP-потоки для web-приложения;
- единую JSON-модель бортовых данных;
- плагины для интеграции;
- bridge к OpenCPN и другим морским инструментам.

Практический вывод: Signal K - лучший кандидат на стандартный интерфейс `NavDataProvider` для будущей версии.

### CANboat

CANboat полезен, если на борту есть NMEA 2000/CAN-сеть. Он помогает декодировать данные NMEA 2000 и превращать их в более пригодный для приложения вид.

Практический вывод: CANboat нужен не пользователю, а интеграционному слою. Его можно указать как технический bridge между физической сетью датчиков и Signal K/нашим приложением.

### GPSD

GPSD закрывает простую задачу: получить GPS/GNSS-данные от приемника через единый интерфейс. Для бортового компьютера это быстрый способ получить позицию без сложной морской сети.

Практический вывод: GPSD можно использовать как самый простой реальный источник координат в полевом прототипе.

## Общие routing engines

### OSRM

OSRM - быстрый routing engine, обычно используемый для дорожной маршрутизации. Он не является водным навигатором, но интересен как пример API и подхода к маршрутизации по графу.

Для нашего кейса OSRM можно рассматривать только если команда строит собственный граф водных/смешанных участков и хочет использовать готовый движок. Но стандартные автомобильные профили OSRM для аэролодки не подходят.

Практический вывод: OSRM полезен как референс API и архитектуры маршрутизатора, но не как готовое решение "водной аэролодочной" навигации.

### Valhalla, GraphHopper

Valhalla и GraphHopper похожи по роли: это routing engines, которые умеют работать с графами и профилями стоимости. Они могут быть полезны, если проекту нужен более промышленный backend маршрутизации.

Ограничение то же: водная проходимость аэролодки, режим нагнетателя и расход топлива все равно потребуют кастомного профиля и своих данных.

### ROS Nav2 и OMPL

ROS Nav2 и OMPL относятся скорее к робототехнике, чем к классической морской навигации. Они полезны, если проект развивается в сторону автономной аэролодки или локального планирования движения.

Что можно взять:

- готовые реализации planner-алгоритмов;
- Hybrid A* через Smac Planner в Nav2;
- RRT/RRT* и другие sampling-based planners через OMPL;
- архитектурное разделение global planner / local planner / costmap.

Что не подходит для MVP:

- тяжелая инфраструктура;
- необходимость модели робота и сенсоров;
- сложность демонстрации для задачи "цифровой штурман".

Практический вывод: ROS/OMPL стоит использовать как источник алгоритмических решений и библиотек для будущей автономности, но не как базу первого хакатонного интерфейса.

## Коммерческие морские приложения и SDK

Navionics/Garmin, C-MAP, Raymarine и другие коммерческие экосистемы дают сильный UX и богатые карты, но для хакатона есть ограничения:

- публичные API/SDK могут быть недоступны или ограничены;
- данные часто лицензируются;
- трудно встроить собственную модель аэролодки;
- зависимость от внешнего продукта может усложнить демо.

Практический вывод: их стоит изучать как UX-референсы, но не делать основой решения, если нет заранее подтвержденного доступа к SDK.

## Сравнение готовых интерфейсов

| Интерфейс | Что дает готовым | Тип интеграции | Насколько закрывает кейс | Ограничения | Рекомендация |
| --- | --- | --- | --- | --- | --- |
| OpenCPN | карты, маршруты, GPS/AIS/NMEA, GPX, плагины | приложение/плагин/экспорт | средне | нет модели аэролодки | использовать как референс и внешний viewer |
| Signal K | JSON-модель судовых данных, сервер, WebSocket/HTTP | middleware/API | высоко для телеметрии | не строит маршрут сам | основной кандидат для данных борта |
| CANboat | декодирование NMEA 2000 | bridge/tooling | средне | нужен CAN/N2K доступ | использовать для будущего железа |
| GPSD | GPS/GNSS данные | локальный сервис/API | средне | только позиция и часть навигации | быстрый полевой GPS |
| NOAA ENC | официальные электронные карты США | данные | средне | регионально ограничено | источник примеров ENC |
| IHO S-57/S-101/S-100 | стандарты гидрографических данных | стандарт | высоко как ориентир | не готовый API | учитывать в data model |
| OpenStreetMap/OpenSeaMap | открытые геоданные | данные | средне | неполнота и качество | использовать осторожно для демо |
| GeoJSON | простые геоданные | формат | высоко для MVP | не оптимален для больших карт | основной формат прототипа |
| GeoPackage | офлайн-геоданные | формат/SQLite | высоко для продукта | сложнее, чем GeoJSON | следующий шаг |
| GPX | маршруты/точки/треки | обмен | высоко | мало семантики риска/топлива | экспорт маршрутов |
| OSRM | routing API по графу | backend/API | низко/средне | не водный по умолчанию | референс, не ядро MVP |
| ROS Nav2 | costmap, planners, Hybrid A* | robotics stack | низко для MVP, высоко для автономности | тяжелый стек | только для будущего autonomous layer |
| OMPL | RRT/RRT* и другие planners | библиотека | низко для MVP, высоко для motion planning | нужна модель состояния | использовать позже |

## Что можно взять готовым

1. **Отображение карты.** MapLibre GL или Leaflet для web-интерфейса, OpenCPN как внешний морской viewer.
2. **Форматы маршрутов.** GPX для обмена с картплоттерами и навигационными приложениями.
3. **Бортовые данные.** Signal K как JSON/API-слой, GPSD для GPS, CANboat для NMEA 2000.
4. **Офлайн-геоданные.** GeoJSON для MVP, GeoPackage/MBTiles для более серьезного продукта.
5. **Алгоритмические библиотеки.** OMPL/Nav2 для будущего автономного планирования, если потребуется.

## Что придется делать самим

1. **Модель проходимости аэролодки.** Готовые морские карты не знают, что аэролодка может пройти там, где обычная лодка не пройдет.
2. **Расход топлива Raptor 650.** Нужны профиль расхода, загрузка, режим нагнетателя и тип поверхности.
3. **Режимы маршрута.** Быстро, экономно, безопасно, с нагнетателем, без нагнетателя.
4. **Объяснение решения.** Пользователю нужно видеть, почему маршрут выбран: экономия топлива, риск мели, короткий проход по мелководью.
5. **Локальные пользовательские данные.** Отметки охотников/рыбаков/операторов: опасные места, проходы, сезонные изменения.
6. **Оценка доверия.** Где данные официальные, где пользовательские, где устаревшие, где предположение.

## Рекомендуемая архитектура интерфейсов

```text
Web UI
  -> Map renderer: MapLibre/Leaflet
  -> Route planner: custom A* core
  -> Data store: GeoJSON now, GeoPackage/MBTiles later

Adapters:
  -> NavDataProvider
      -> ManualInput
      -> GPSD
      -> Signal K
  -> MarineTelemetryProvider
      -> Signal K
      -> CANboat/NMEA 2000
      -> NMEA 0183
  -> RouteExporter
      -> GPX
      -> GeoJSON
      -> OpenCPN-compatible workflow
```

Ключевой принцип: готовые интерфейсы должны быть вокруг расчетного ядра, а не внутри него. Тогда можно менять источник данных или карту, не переписывая алгоритм маршрута.

## UX-функции, которые стоит подсмотреть у готовых навигаторов

- route overview до старта;
- список waypoints;
- ETA и distance remaining;
- cross-track error, если есть трек движения;
- предупреждения о глубине, препятствиях и ограничениях;
- запись трека;
- экспорт/импорт GPX;
- режим "следовать по маршруту";
- индикатор качества GPS;
- понятный статус источников данных.

Для аэролодки нужно добавить уникальные UX-блоки:

- "режим нагнетателя";
- "проходимость участка";
- "расход топлива по участкам";
- "риск застревания/мели";
- "почему маршрут отличается от обычного водного".

## Итоговая рекомендация

Для хакатона оптимальная комбинация:

1. Свой расчет маршрута на A*.
2. Карта и зоны в GeoJSON.
3. Экспорт маршрута в GPX.
4. Внутренняя модель состояния борта, совместимая по смыслу с Signal K.
5. В документации показать будущие адаптеры Signal K, GPSD, CANboat, NMEA 0183/2000.
6. OpenCPN использовать как референс и потенциальный внешний просмотрщик, но не как единственное ядро решения.
