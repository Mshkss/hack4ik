# 09. Сценарии демонстрации

## Зачем этот блок

Сценарии нужны, чтобы команда не просто "показала приложение", а доказала, что прототип закрывает требования кейса: offline-first, карта как данные, расчёты, разные режимы, предупреждения и влияние параметров лодки.

## Сценарий 1. Подготовка карты и автономная работа

Пользователь открывает приложение и выбирает заранее загруженную карту.

Что показать:

- карта доступна локально;
- есть разметка поверхностей;
- есть ограничения движения;
- есть коэффициенты расчёта;
- приложение продолжает работать без сети.

Критерий успеха: эксперт понимает, что данные не подтягиваются с сервера при каждом расчёте.

## Сценарий 2. Построение маршрута из точки A в точку Б

Пользователь выбирает старт и финиш, задаёт параметры Raptor 650, запас топлива, загрузку и режим.

Что показать:

- маршрут построен системой;
- указаны длина, время, расход, остаток топлива и запас хода;
- рискованные участки подсвечены;
- есть короткое объяснение выбора маршрута.

Критерий успеха: маршрут не задан вручную как картинка, а считается из данных.

## Сценарий 3. Сравнение разных режимов движения

Для одной пары точек строятся быстрый, экономичный, кратчайший и безопасный маршруты.

Что показать:

- маршруты отличаются;
- показатели отличаются;
- эксперт видит компромиссы;
- если два маршрута совпали, система объясняет почему.

Критерий успеха: режимы влияют на расчёт, а не только меняют цвет линии.

## Сценарий 4. Учёт сложного участка и предупреждений

Маршрут проходит через мелководье, шугу, траву, камни, болото или смешанную зону.

Что показать:

- участок повышает расход/риск;
- может теряться глиссирование;
- система предупреждает пользователя;
- при возможности предлагается альтернативный путь.

Критерий успеха: карта влияет на итоговый маршрут и карточку риска.

## Сценарий 5. Влияние конфигурации лодки и загрузки

Пользователь сравнивает Raptor 650 без поддува и с поддувом, а также меняет загрузку.

Что показать:

- один маршрут может стать допустимым с поддувом;
- полная загрузка повышает расход;
- запас хода меняется;
- риск или потеря глиссирования меняются.

Критерий успеха: параметры лодки участвуют в расчёте.

## Сценарий 6. Карточка маршрута

После построения пользователь открывает итоговую карточку.

Что показать:

- карта/схема прохождения;
- режим движения;
- длина;
- время;
- расход топлива;
- остаток топлива;
- запас хода;
- рисковые участки;
- предупреждения;
- объяснение выбора маршрута.

Критерий успеха: карточку можно показать как итог решения эксперту или экипажу.

## Мини-чеклист демо

- [ ] Приложение запускается локально.
- [ ] Есть заранее подготовленная карта.
- [ ] Есть разметка типов поверхности.
- [ ] Есть параметры Raptor 650.
- [ ] Строится маршрут A-Б.
- [ ] Сравниваются минимум 4 режима.
- [ ] Есть предупреждения.
- [ ] Есть карточка маршрута.
- [ ] Есть объяснение расчётной логики.
- [ ] Есть схема будущего переноса в бортовой терминал.
